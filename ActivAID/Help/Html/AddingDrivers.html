Adding a New Driver

These steps describe how to add a new driver to the ActivATE™ environment.

  1. To get started, use one of the following methods:
  1. Right-click anywhere in the tree view in the Configuration pane and select "Add New" from the context menu. 


     
  2. Drag and drop the .DLL or .EXE assembly file that contains the driver you would like to add to the tree view in the Configuration pane.  If the assembly is recognized as an ActivATE™-specific assembly, the ActivATE™ environment will try to automatically add the driver for you.  Simply provide a test program reference name and a description in the fields as shown below, then click OK:



    If the assembly is not recognized, the configuration tool will appear instead.  Follow the instructions below to continue adding the driver.
  1. The configuration tool appears.  Click the file browse button next to the File Name text box to open a file browser dialog and select the driver assembly to add.  The driver assembly should be a .NET or COM DLL or EXE file. 
     
  2. After the driver assembly has been selected, the classes that are available within the assembly will be displayed in the Assembly Classes tree view.  Expand a class to view the public methods and properties that are exposed in that class.
     
  3. The tool will attempt to recommend the driver and UI (User Interface) classes to use.  The driver class will be used to create an object for the test program to access, and its public API's will be available to the test program.  The UI class is optional, but can only be specified for an ActivATE™-specific assembly.  If provided, the UI class determines the form that will be created when the user clicks the icon for the driver on the Outlook-style toolbar.  A UI provides a quick and easy way of accessing the driver's functionality without having to run a test program. 

    If the assembly is a recognized ActivATE™-specific assembly, the tool should successfully locate the correct classes.  These will be highlighted in yellow in the Assembly Classes tree view and they will be automatically filled in in the Driver Class Name and UI Class Name text boxes, as shown below:

     
    The GPIBDriver.dll assembly is an ActivATE™-specific assembly and the tool was therefore able to recommend the correct classes we should use for this driver.


    If the tool was not able to recommend any classes, or if you would like to change the selected classes, right-click on the class you would like to use in the tree view and select either "Use as Driver class" or "Use as UI class" from the context menu.


     

  4. Enter the name to be used to reference this driver from the test program in the TP Reference text box.  This name must be unique among all other test program reference names within the same configuration.
     
  5. In the ToolBar Group text box, enter the group name that this driver will appear under in the Outlook-style toolbar.  This field only applies if a UI class has been selected.  The default group is "Devices".  Additional groups will be added after the Devices pane in the Outlook-style toolbar.  The following example shows the result of adding a device in a group called "Switch Cards":


     
  6. Enter a brief description of the driver in the Description text box.  This field is optional. 
     
  7. Finally, click the Add Driver button to add the new driver to the current configuration, or click Cancel to exit without adding the driver.

 

Important:  Committing Changes

After the new driver has successfully been added, the Reinitialize button on the toolbar will be highlighted in yellow as shown below.  The new driver will NOT be available until the environment has been reinitialized or until the application is restarted.  To reinitialize, make sure all test programs and driver UI's are closed and then click the Reinitialize button.

If at any time you wish to cancel your changes, click the Rollback button at the bottom of the properties page.  This will undo all changes that have been made since the last reinitialization.  The Rollback button is only enabled if there are uncommitted changes (until the system is reinitialized or until the application is shut down).
